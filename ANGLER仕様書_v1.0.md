# ANGLER 仕様書 v1.0

## 1. アプリ概要

### アプリ名

**ANGLER**

### アプリ種別

スマホで開けるWebアプリ。

スマートフォンのブラウザから使用でき、アプリストアへの公開やインストールを前提としない。

クラウド上の静的ホスティングに公開し、外出先からURLでアクセスして使用できる。

### 主な目的

釣り用リールにラインを巻く際の、下巻き量・メインライン量・ハンドル回転数を計算し、リールとスプールごとに保存する。

これにより、次回以降のライン巻き替え時に、過去の設定を呼び出して再利用できるようにする。

---

## 2. 作成目的

現在、下巻き計算にはスマホアプリ「Lure Bank」を使用しているが、以下の課題がある。

### 現状の困りごと

- 複数のリールを持っているため、毎回条件を入力して計算する必要がある
- 計算結果を保存するメモ機能がない
- どのリール・どのスプールに、どのラインを何m巻いたか管理しづらい
- ラインを巻き替えるたびに、過去の下巻き量を思い出す必要がある
- 下巻き量をmで計算できても、実際の作業ではハンドル何回転分かが分かりづらい

### ANGLERで解決したいこと

- リールごとにスプールを管理する
- スプールごとに現在のライン設定を保存する
- 下巻き量を計算する
- 下巻き量をハンドル回転数に変換する
- 実際に巻いた結果を履歴として保存する
- 次回のライン巻き替え時に、過去の設定をすぐ確認できるようにする

## 4. 対象ユーザー

ANGLERは、以下のようなユーザーを想定する。

- バス釣り、エギングなど複数ジャンルの釣りをする人
- スピニングリールやベイトリールを複数所有している人
- 替えスプールを使ってラインを使い分けている人
- PE、フロロ、ナイロン、エステルなど複数のラインを使う人

- ライン巻き替えの履歴をスマホで管理したい人

## 5. アプリの基本コンセプト

ANGLERは、単なる計算アプリではなく、

**一度計算した条件を保存し、次回から迷わないためのライン巻き管理アプリ**

## 8. ライン径データベース仕様

### 8.1 対象ライン

アプリ内に、ライン径の換算表をデータベースとして持たせる。

対象ラインは以下とする。

| ライン種別 | 登録項目 |
|---|---|
| ナイロン | 号数、kg、lb、太さmm |
| フロロ | 号数、kg、lb、太さmm |
| エステル | 号数、kg、lb、太さmm |
| PE | 号数、kg、lb、太さmm |

---

### 8.2 データベースの目的

ライン径データベースは、以下の目的で使用する。

- 号数を選ぶと、太さmmを自動入力する
- lbを選ぶと、太さmmを自動入力する
- PE号数を選ぶと、太さmmを自動入力する
- ライン表示名には、号数とlbを併記する
- 下巻き計算で使うライン直径を決定する
- 入力ミスを減らす
- よく使うライン登録を簡単にする

---

### 8.3 計算で使う値

下巻き計算では、最終的に **太さmm** を使用する。

```text
ライン容量 ≒ ライン長さ × ライン直径²
```

号数、kg、lbは入力補助・表示用として扱い、計算には太さmmを使用する。

---

### 8.4 手動補正

ライン径はメーカーや銘柄によって差があるため、自動入力された太さmmは手動で修正できるようにする。

#### 例

```text
フロロ 6lb
標準太さ：0.205mm
使用ラインの実表記：0.200mm

→ 0.200mmに手動変更可能
```

---

## 9. 下巻き量計算仕様

### 9.1 基本的な考え方

スプールのライン容量は、ラインの長さとライン直径の2乗に比例すると考える。

#### 基本式

```text
ライン容量 ≒ ライン長さ × ライン直径²
```

---

### 9.2 スプール容量の計算

カタログ表記の糸巻き量から、スプールの基準容量を計算する。

```text
スプール容量 = 基準ライン長さ × 基準ライン直径²
```

#### 例

```text
基準ライン：PE0.6号
基準ライン長さ：150m
基準ライン直径：0.132mm

スプール容量 = 150 × 0.132²
```

---

### 9.3 メインライン容量の計算

```text
メインライン容量 = メインライン長さ × メインライン直径²
```

---

### 9.4 残り容量の計算

```text
残り容量 = スプール容量 - メインライン容量
```

---

### 9.5 下巻き量の計算

```text
下巻き長さ = 残り容量 ÷ 下巻きライン直径²
```

---

### 9.6 注意点

この計算は目安であり、実際の巻き量は以下の条件で変わる可能性がある。

- ラインメーカーによる実径の違い
- ラインの硬さ
- ラインの潰れ方
- 巻きテンション
- スプール形状
- PEラインとフロロ・ナイロンの違い
- 結び目やテープの厚み

そのため、計算結果は絶対値ではなく目安として扱う。

---

## 10. ハンドル回転数計算仕様

### 10.1 基本方針

下巻き量をmで表示するだけでなく、ハンドル回転数にも変換する。

これにより、実際の巻き作業で「何回転巻けばよいか」を確認できる。

---

### 10.2 最大巻取り長の使い方

リールメーカーが公表している最大巻取り長は、基本的にスプールにラインが十分巻かれた状態の数値である。

ANGLERでは、最大巻取り長をそのまま下巻き時の1回転巻取り長としては使わない。

代わりに、最大巻取り長とギア比から満巻き時の有効スプール径を逆算し、空スプール径と組み合わせて下巻き中の平均スプール径を推定する。

```text
満巻き時スプール径mm = 最大巻取り長cm × 10 ÷ 円周率 ÷ ギア比
```

下巻きを開始する空スプール状態では、スプール径が小さいため、1回転あたりの巻取り長も短くなる。

#### 誤差の例

```text
最大巻取り長：72cm
空スプール付近の実巻取り長：50cm前後

45m ÷ 0.72m = 約63回転
45m ÷ 0.50m = 約90回転
```

このように、最大巻取り長をそのまま使うと大きな誤差が出る可能性がある。

そのため、ANGLERでは最大巻取り長を満巻き時スプール径の逆算に使い、下巻き時の巻取り長は平均スプール径から計算する。

---

### 10.3 スプール径から巻取り長を計算する

ハンドル1回転でローターが何回転するかは、リールのギア比で決まる。

スプール外周は以下で求める。

```text
スプール外周 = スプール直径 × 円周率
```

ハンドル1回転あたりの巻取り長は、以下で求める。

```text
1回転あたりの巻取り長 = スプール外周 × ギア比
```

つまり、

```text
1回転あたりの巻取り長 = スプール直径 × 円周率 × ギア比
```

となる。

---

### 10.4 下巻き中のスプール径変化

下巻きを巻いていくと、スプール径は少しずつ太くなる。

そのため、空スプール径だけで計算すると、実際より回転数が多めに出る可能性がある。

より現実に近い計算では、空スプール径と下巻き終了時の推定径の平均を使う。

空スプール径はノギスなどがないと測定しづらいため、初期版では任意入力とする。
未入力の場合は、最大巻取り長とギア比から逆算した満巻き時スプール径に一定係数を掛けた推定値を使う。

```text
推定空スプール径 = 22mm
```

測定値が入力されている場合は、入力値を優先する。

```text
平均スプール径 = (空スプール径 + 下巻き終了時の推定径) ÷ 2
```

```text
平均巻取り長 = 平均スプール径 × 円周率 × ギア比
```

```text
ハンドル回転数 = 下巻き量 ÷ 平均巻取り長
```

---

### 10.5 初期版での回転数計算

初期版では、以下の情報を使って回転数を計算する。

| 使用項目 | 内容 |
|---|---|
| 下巻き量 | 下巻き計算で求めた長さ |
| 空スプール径 | ラインを巻いていない状態の直径。未入力時は推定値 |
| ギア比 | リールのギア比 |
| 下巻き終了時の推定径 | 補正用 |
| 平均スプール径 | 回転数計算に使用 |

---

### 10.6 作業画面の表示例

```text
下巻きライン：ナイロン12lb
必要量：約45m
推定1回転巻取り長：約55cm
ハンドル回転数：約82回転

作業手順：
1. 下巻きラインを結ぶ
2. ハンドルを約82回転させる
3. メインライン PE0.6号を150m巻く
4. 巻き終わりを確認する
5. 評価を保存する
```

## 11. 評価・補正メモ仕様

計算結果と実際の巻き上がりには差が出るため、巻き終わりの評価を保存できるようにする。

### 評価項目

| 評価 | 内容 |
|---|---|
| ちょうど良い | 次回も同じ条件でよい |
| 少ない | 次回は下巻きを増やす |
| 多い | 次回は下巻きを減らす |

### 補正メモ例

```text
82回転だと少し少なかった。
次回は＋5回転くらい。
```

```text
90回転だと多かった。
次回は85回転にする。
```

## 12. データ保存仕様

### 12.1 保存方式

初期版では、以下の方式を採用する。

```text
スマホ内保存
＋
バックアップ出力
＋
バックアップ読込
```

---

### 12.2 スマホ内保存

通常使用時は、スマホのブラウザ内にデータを保存する。

クラウド公開後も、登録データはサーバーに保存せず、アクセスした端末のブラウザ内に保存する。

#### 候補

- LocalStorage
- IndexedDB

初期版では、実装が簡単な **LocalStorage** を候補とする。

データ件数が増えたり、画像保存が必要になった場合は **IndexedDB** への変更を検討する。

---

### 12.3 スマホ内保存のメリット

| メリット | 内容 |
|---|---|
| ログイン不要 | アカウント登録なしで使える |
| 作成が簡単 | 初期版を作りやすい |
| 通信不要 | オフラインでも使いやすい |
| 個人管理向き | 自分だけのリール管理に向いている |

---

### 12.4 スマホ内保存の注意点

| 注意点 | 内容 |
|---|---|
| 機種変更 | データ移行が必要 |
| ブラウザ変更 | SafariとChromeでは保存先が別になる |
| データ削除 | ブラウザのサイトデータ削除で消える可能性がある |
| 複数端末共有 | 自動同期はできない |

---

### 注意点

クラウド公開は、アプリ本体を外出先から開けるようにするための対応である。

初期版ではクラウド同期を行わないため、スマホとPCなど複数端末で自動的に同じデータを共有することはできない。

端末を変える場合は、JSONバックアップ出力と読込でデータを移行する。

## 18. 開発方式の候補

ANGLERはスマホ対応Webアプリとして作成する。

### 候補A：HTML / CSS / JavaScriptのみ

| 項目 | 内容 |
|---|---|
| メリット | シンプルで作りやすい |
| メリット | スマホブラウザで動く |
| メリット | LocalStorageを使いやすい |
| デメリット | 複雑になると管理しづらい |

### 候補B：Reactなどのフロントエンドフレームワーク

| 項目 | 内容 |
|---|---|
| メリット | 画面管理がしやすい |
| メリット | アプリ風のUIにしやすい |
| デメリット | 初期設定が少し難しい |

### 候補C：Python Flask

| 項目 | 内容 |
|---|---|
| メリット | Pythonで処理を書きやすい |
| メリット | PCでの開発に向いている |
| デメリット | スマホ単体で使うにはサーバーが必要 |

### 初期版のおすすめ

初期版は、以下の構成がよい。

```text
HTML
＋
CSS
＋
JavaScript
＋
LocalStorage
```

理由は、スマホで開けるWebアプリとして作りやすく、ログイン不要・サーバー不要で運用しやすいため。

## 19. 仕様まとめ

ANGLERは、釣り用リールのライン巻き替えを管理するWebアプリである。

主な特徴は以下。

- スマホで開ける
- リールを選んでからスプールを選ぶ
- スプールごとにライン設定を保存する
- ライン径データベースから太さmmを呼び出す
- 下巻き量を計算する
- 下巻き量をハンドル回転数に変換する
- 空スプール径または推定空スプール径とギア比を使って、回転数の誤差を減らす
- 実際に巻いた回転数と評価を保存する
- バックアップ出力・読込で機種変更に対応する

初期版では、まず個人利用に必要な機能に絞り、後からクラウド保存やCSV出力などを追加できる設計とする。
